## 7.1 六大设计原则

#### 1. 单一职责原则 SRP (Single Responsibility)
- 定义：一个类/函数只有一个引起它变化的原因，只承担一项职责。
- 核心：职责 = 变化的轴线；多职责意味着一处改动波及多个无关需求。
- 违反示例：UserService 既负责持久化又负责发邮件又负责权限校验。
- 修正：拆为 UserRepository / EmailService / AuthService。
- 应用阶段：概要设计（模块划分）+ 详细设计（类/函数划分）。

#### 2. 开闭原则 OCP (Open-Closed)
- 定义：对扩展开放，对修改关闭；新增行为通过扩展而非改老代码实现。
- 核心：抽象 + 多态隔离变化点；改老代码 = 引入回归风险。
- 违反示例：新增支付方式时改 PayService 的 switch 分支。
- 修正：定义 PaymentStrategy 接口，新增方式 = 新增实现类。
- 应用阶段：概要设计（识别扩展点）+ 详细设计（抽象接口设计）。

#### 3. 里氏替换原则 LSP (Liskov Substitution)
- 定义：所有引用父类的地方必须能透明地使用子类对象，行为不破坏。
- 核心：子类不得加强前置条件、削弱后置条件、抛出新的异常类型。
- 违反示例：子类重写 width/height setter 后，setWidth 再 setHeight 不再保持正方形面积语义。
- 修正：子类遵守父类契约；契约不符则不该继承，改用组合。
- 应用阶段：详细设计（继承关系与契约约束）。

#### 4. 接口隔离原则 ISP (Interface Segregation)
- 定义：客户端不应被迫依赖它不使用的方法；接口按消费方需要拆分。
- 核心：胖接口拆为多个细粒度接口，实现类按需实现。
- 违反示例：一个 Worker 接口含 work/eat/sleep，机器人被迫实现 eat/sleep 抛异常。
- 修正：拆为 IWorkable / IEatable / ISleepable，机器人只实现 IWorkable。
- 应用阶段：详细设计（接口拆分）。

#### 5. 依赖倒置原则 DIP (Dependency Inversion)
- 定义：高层模块不依赖低层模块，二者均依赖抽象；抽象不依赖细节，细节依赖抽象。
- 核心：面向接口编程，通过依赖注入解耦。
- 违反示例：OrderService 直接 new MysqlRepository，换存储要改 OrderService。
- 修正：定义 IRepository 接口，OrderService 依赖接口，实现类注入。
- 应用阶段：概要设计（模块间依赖方向）+ 详细设计（依赖注入）。

#### 6. 迪米特法则 LoD (Law of Demeter / 最少知识原则)
- 定义：一个对象应尽可能少地了解其他对象的内部结构；只与"直接朋友"通信。
- 核心：不链式调用陌生对象的内部（a.b.c().d()），降低耦合。
- 违反示例：order.getCustomer().getAddress().getCity() 链式穿越。
- 修正：让 Order 提供 getCustomerCity() 方法，封装内部结构。
- 应用阶段：详细设计（方法间调用与封装边界）。

## 7.2 创建型模式（5 个）

#### 单例模式 Singleton
- 意图：保证一个类只有一个实例，提供全局访问点。
- 适用：配置中心、连接池、日志器等共享资源。
- 结构：私有构造 + 类级静态实例 + 静态获取方法。
- 反模式：滥用作全局变量；多线程下未做双重检查锁。
- 关联：与享元都做共享，但享元共享细粒度对象，单例共享唯一实例。

#### 工厂方法模式 Factory Method
- 意图：定义创建对象的接口，由子类决定实例化哪个类。
- 适用：创建对象时不想指定确切类、需将创建与使用解耦。
- 结构：Creator 声明工厂方法，ConcreteCreator 返回 ConcreteProduct。
- 反模式：用一个大 if/else 集中创建所有产品（那应升级为抽象工厂）。
- 关联：抽象工厂是工厂方法的升级，创建产品族；建造者关注分步构建。

#### 抽象工厂模式 Abstract Factory
- 意图：创建一系列相关或相互依赖的对象（产品族），无需指定具体类。
- 适用：跨主题/平台的 UI 组件族、多数据库驱动族。
- 结构：AbstractFactory 声明多个创建方法，ConcreteFactory 实现一族产品。
- 反模式：产品族扩展困难（新增产品类型要改所有工厂）。
- 关联：工厂方法创建单一产品，抽象工厂创建产品族。

#### 建造者模式 Builder
- 意图：将复杂对象的构造与其表示分离，同样构造过程可造不同表示。
- 适用：参数多、可选参数多、需分步构建的对象（如 SQL/配置/文档）。
- 结构：Director 编排构建步骤，Builder 定义各步骤接口，ConcreteBuilder 实现。
- 反模式：用几十个参数的构造函数 + 大量 null 占位。
- 关联：工厂方法一次性创建，建造者分步创建；常与不可变对象配合。

#### 原型模式 Prototype
- 意图：通过克隆现有实例创建新对象，而非重新构造。
- 适用：创建成本高（需查库/计算）、或需保留某时刻状态副本。
- 结构：Prototype 声明 clone 方法，子类实现深/浅拷贝。
- 反模式：浅拷贝共享可变内部对象导致串改。
- 关联：与享元都涉及对象复用；原型是克隆，享元是共享。

## 7.3 结构型模式（7 个）

#### 适配器模式 Adapter
- 意图：将一个类的接口转换成客户端期望的另一个接口，使不兼容类协同。
- 适用：复用已有类但接口不匹配、对接第三方库。
- 结构：Adapter 实现 Target 接口，内部持有 Adaptee 并委托调用。
- 反模式：为每个调用硬编码转换逻辑散落各处。
- 关联：装饰器增强不改接口，适配器转换接口；外观简化复杂子系统。

#### 装饰器模式 Decorator
- 意图：动态地给对象增加职责，不改变其接口且不依赖继承。
- 适用：可叠加的功能（日志/缓存/压缩/校验）、IO 流包装。
- 结构：Decorator 实现 Component 接口并持有 Component 引用，调用前后增强。
- 反模式：用继承堆出大量子类组合各种功能。
- 关联：适配器改接口，装饰器保接口；代理控制访问，装饰器增强行为。

#### 代理模式 Proxy
- 意图：为其他对象提供代理以控制访问，附加延迟加载/权限/远程等控制。
- 适用：远程代理、虚拟代理（懒加载）、权限代理、缓存代理。
- 结构：Proxy 与 RealSubject 实现同一接口，Proxy 控制对 RealSubject 的访问。
- 反模式：代理与真实对象职责混淆，代理承担业务逻辑。
- 关联：装饰器增强行为，代理控制访问；结构相似动机不同。

#### 外观模式 Facade
- 意图：为复杂子系统提供一个统一的高层接口，简化客户端调用。
- 适用：封装复杂子系统、分层架构的入口、简化第三方 API。
- 结构：Facade 持有子系统组件引用，提供粗粒度方法组合子系统调用。
- 反模式：Facade 变成"上帝对象"承载所有业务逻辑。
- 关联：适配器转换接口，外观简化接口；中介者协调对象间交互。

#### 桥接模式 Bridge
- 意图：将抽象部分与实现部分分离，使二者可独立变化。
- 适用：跨维度变化（如形状×颜色、消息类型×发送渠道）。
- 结构：Abstraction 持有 Implementor 引用，二者各自独立扩展。
- 反模式：用多层继承爆炸组合（Shape×Color 全排列子类）。
- 关联：策略模式将算法分离，桥接将实现分离；前者运行时切换，后者结构解耦。

#### 组合模式 Composite
- 意图：将对象组合成树形结构表示"部分-整体"层次，客户端统一对待单个与组合。
- 适用：树形结构（文件系统/UI 组件/组织架构）、需统一处理叶子和容器。
- 结构：Component 声明统一接口，Leaf 和 Composite 都实现它，Composite 持有子组件列表。
- 反模式：客户端用 instanceof 区分叶子和容器再分别处理。
- 关联：与装饰器都递归组合；装饰器单向包装，组合是树形。

#### 享元模式 Flyweight
- 意图：共享细粒度对象以减少内存占用，将内部状态共享、外部状态外置。
- 适用：大量相似对象（棋子/字符/缓存键）、内存敏感场景。
- 结构：FlyweightFactory 维护享元池，Flyweight 存内部状态，外部状态由客户端传入。
- 反模式：把可变状态也放进享元导致并发串改。
- 关联：单例共享唯一实例，享元共享同类多实例；与原型都涉及对象复用。

## 7.4 行为型模式（11 个）

#### 策略模式 Strategy
- 意图：定义算法族，各自封装，使可互换。
- 适用：多种算法择一、消除条件分支、运行时切换算法。
- 结构：Context 持有 Strategy 接口引用，委托具体策略执行。
- 反模式：用 if/else 或 switch 堆叠算法选择，每加一个算法改一处。
- 关联：与状态模式结构相同，动机不同（策略由外部选择，状态由内部流转）。

#### 模板方法模式 Template Method
- 意图：在父类定义算法骨架，将某些步骤延迟到子类实现。
- 适用：固定流程但个别步骤可变、复用算法结构。
- 结构：AbstractClass 用模板方法编排步骤，子类重写可变步骤。
- 反模式：子类重写了不该重写的步骤破坏算法骨架。
- 关联：策略用组合切换算法，模板方法用继承固定骨架。

#### 观察者模式 Observer
- 意图：一对多依赖，对象状态变更时自动通知所有观察者。
- 适用：事件订阅、消息广播、UI 与数据解耦。
- 结构：Subject 维护观察者列表，变更时遍历调用 update。
- 反模式：被观察者直接调用各具体观察者的特有方法。
- 关联：与发布订阅相似；中介者解决多对多交互，观察者解决一对多通知。

#### 迭代器模式 Iterator
- 意图：提供一种方法顺序访问聚合对象内部元素，不暴露其内部结构。
- 适用：统一遍历不同容器、支持多种遍历方式、延迟遍历。
- 结构：Aggregate 创建 Iterator，Iterator 实现 hasNext/next 接口。
- 反模式：客户端直接访问容器内部数组按下标遍历。
- 关联：常与组合模式配合遍历树形结构。

#### 责任链模式 Chain of Responsibility
- 意图：将请求沿处理者链传递，每个处理者决定处理或传给下一个。
- 适用：多级审批、过滤器链、中间件、多种处理者择一。
- 结构：Handler 持有下一个 Handler 引用，决定处理或转发。
- 反模式：链无限循环；处理者既处理又转发导致重复处理。
- 关联：与中介者都解耦发送者与接收者；责任链单向传递，中介者集中协调。

#### 命令模式 Command
- 意图：将请求封装为对象，使可参数化、排队、记录、撤销。
- 适用：撤销/重做、任务队列、事务、宏操作。
- 结构：Command 声明 execute/undo 接口，ConcreteCommand 持有接收者。
- 反模式：命令对象承载过多业务逻辑变成"上帝命令"。
- 关联：与策略都封装行为；策略聚焦算法择一，命令聚焦请求封装与撤销。

#### 备忘录模式 Memento
- 意图：在不破坏封装的前提下捕获对象内部状态，以便日后恢复。
- 适用：撤销机制、检查点、事务回滚。
- 结构：Originator 创建/恢复 Memento，Caretaker 保管 Memento 不读其内容。
- 反模式：Caretaker 直接读写 Memento 内部状态破坏封装。
- 关联：常与命令模式配合实现撤销（命令产生备忘录）。

#### 状态模式 State
- 意图：对象状态改变时其行为也随之改变，看起来像换了类。
- 适用：状态机、订单/流程流转、对象多种状态行为差异大。
- 结构：Context 持有 State 引用，State 定义状态行为接口，ConcreteState 实现各状态行为。
- 反模式：用一个 state 字段 + 大量 if/else 分支不同状态行为。
- 关联：与策略结构相同，动机不同（状态由内部流转驱动，策略由外部选择）。

#### 访问者模式 Visitor
- 意图：在不改变各元素类的前提下定义作用于其上的新操作。
- 适用：稳定对象结构但操作频繁变化（如编译器 AST、报表导出）。
- 结构：Visitor 声明 visit 方法重载各 Element，Element 用 accept(visitor) 双重分派。
- 反模式：对象结构频繁变化时每加一个 Element 要改所有 Visitor。
- 关联：与迭代器都遍历结构；迭代器不关心操作，访问者封装操作。

#### 中介者模式 Mediator
- 意图：用一个中介对象封装一组对象的交互，使对象间不直接引用而解耦。
- 适用：多组件交互复杂（聊天室/表单联动/事件总线）、GUI 组件协作。
- 结构：Mediator 定义交互接口，ConcreteMediator 协调各 Colleague，Colleague 只与 Mediator 通信。
- 反模式：中介者变成"上帝对象"承载所有逻辑；Colleague 之间仍直接引用。
- 关联：与观察者都解耦对象交互；观察者一对多广播，中介者多对多集中协调；与外观都封装子系统，外观单向简化，中介者双向协调。

#### 解释器模式 Interpreter
- 意图：给定一种语言，定义其文法表示和一个解释器解释该语言中的句子。
- 适用：规则引擎、表达式求值、配置/DSL 解析、正则引擎。
- 结构：AbstractExpression 声明 interpret，TerminalExpression/NonterminalExpression 实现文法规则。
- 反模式：文法复杂时类爆炸；简单需求硬套解释器过度设计。
- 关联：常与组合模式配合构建语法树（NonterminalExpression 持有子表达式）。
