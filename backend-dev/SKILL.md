# 后端项目开发规范

用于在 Python 后端项目中编写、修改、评审代码时，保持开发行为一致，特别适用于使用 uv 包管理器、Tortoise ORM 的后端项目。

## 何时使用

- 新增或修改后端代码。
- 评审代码变更。
- 与用户讨论技术方案选型。
- 执行重构或清理代码。
- 不确定是否该做某处修改时。

---

## 📝 注释与文档要求

### 解释 "为什么" 而非 "是什么"

代码本身已经说明了操作，注释需要重点补充背后的业务逻辑、设计决策或技术妥协（Trade-offs）。

```python
# ❌ 坏示例：解释代码在做什么（代码本身已说明）
# 将 i 加 1
i += 1

# ✅ 好示例：解释为什么这么做
# 从 1 开始计数是因为数据库自增 ID 从 1 起，0 被保留为占位符
i = 1
```

### 保持精简与必要

不需要为自解释代码添加冗余注释。

```python
# ❌ 不需要注释
user_count = len(users)  # 获取用户数量

# ✅ 不需要注释（自解释）
user_count = len(users)
```

### 关注数据结构与接口

在核心服务类和 API 接口中，清晰标注输入输出参数的结构和含义：

```python
class UserService:
    async def create_user(
        self,
        data: UserCreate,  # {username: str, email: str, role: str | None}
    ) -> UserProfile:  # {user_id: int, username: str, email: str, created_at: datetime}
        """
        创建用户并返回用户档案。

        为什么返回完整 Profile 而非仅 user_id：
        前端注册流程需要立即展示用户头像和昵称，减少一次额外请求。
        """
        ...
```

## 🎯 最小改动原则

### 精准修改

只修改与当前任务直接相关的文件和代码行，像外科手术一样精准。

```python
# 场景：修复 get_user 的 N+1 查询问题
# ❌ 坏示例：顺手把整文件的引号统一、变量重命名、函数重排全做了
# ✅ 好示例：只在 get_user 函数内添加 select_related，其余不动
```

### 尊重历史代码

即使现有的代码风格看起来 "不够优雅"，只要它能正常工作且与当前任务无关，请不要顺手重构它。

> 如果看到明显坏味道但任务无关：记在心里，或询问用户是否需要单独处理，不要顺手改。

### 避免大面积格式化

不要在修改几行代码时，触发整个文件的重新格式化，以免干扰版本控制和代码审查。

```python
# ❌ 坏示例：改了 3 行，但格式化工具把 200 行全改了
# ✅ 好示例：只提交与任务相关的改动；格式化另开 PR
```

## 💬 优先询问与沟通

### 拒绝盲目猜测

遇到需求模糊、缺少关键上下文或存在多种平行的技术路径时，先停下来向用户提问。

> ❌ "我觉得你可能想要……所以我做了 A、B、C。"
> ✅ "需求中关于 X 的部分有两种理解：
>    - 理解 A：……，实现较简单
>    - 理解 B：……，扩展性更好
>    您倾向哪种？"

### 提供选择题而非问答题

在提问时，尽量给出几个可行的选项及各自的优缺点：

```
方案 A：直接修改现有表结构
- 优点：改动小，快
- 缺点：历史数据迁移复杂

方案 B：新建关联表，保留原表
- 优点：零迁移风险，可回滚
- 缺点：查询需要 join，稍复杂

您倾向哪个？
```

### 确认高危操作

在执行以下操作前，必须先征求用户同意：

- 大规模重构（影响超过 3 个文件或 100 行）
- 删除核心代码（models、services、关键配置）
- 覆盖重要文件（.env、配置文件、迁移脚本）

### 确认未明示需求

对于用户没有明确说明的功能增加、行为修改或接口调整，必须先说明影响并征求用户意见，确认后再实施：

> "我注意到实现 X 时，需要顺带修改 Y 的行为（原来是 A，会变成 B）。这是否可以接受？"

## 🔄 及时推倒重来（抛弃历史包袱）

### 以当前想法为唯一目标

用户的想法随时可能迭代，始终以用户最新的指示为准。

> 用户说 "之前那个方案不用了，我们换个思路" → 立即停止当前方向，不受之前代码束缚。

### 果断重置

如果用户提出了全新的思路，或者当前的尝试已经陷入死胡同，必须立刻抛弃上下文中的旧想法和失败方案。

```python
# ❌ 坏示例：硬把新需求和旧代码缝合在一起
# ✅ 好示例：先问 "是否废弃之前的方案，全新重写？"，确认后干净重写
```

### 不被对话束缚

不要因为 "前面几轮对话是这么写的" 就强行保留或缝合代码，该推翻重写时就大胆重新开始。

---

## 🌊 灵活处理（务实与变通）

### 打破死板教条

以上所有规则都是为了更好地服务开发，而非不可逾越的铁律。请根据具体场景和用户的语气灵活调整应对策略。

### 听从临时指令

如果用户明确表示以下话语，请立即放下所有的条条框框，以最高效率满足用户当下的需求：

- "先不管规范，赶紧跑通"
- "随便写个草稿"
- "写死数据"
- "先 hack 一下"
- " quickest way"

### 适应用户节奏

- 用户倾向于一步步慢慢来 → 提供渐进式的代码，每步可验证。
- 用户想要一次性完整的解决方案 → 直接给出全貌，但关键决策点仍需确认。

## 🔧 命令规范（uv 包管理器）

本项目统一使用 `uv` 作为包管理器，禁止使用 `pip`、`pipenv`、`poetry` 等其他工具。

| 操作 | 命令 |
|------|------|
| 运行项目 | `uv run --no-sync python main.py` |
| 安装依赖 | `uv sync` |
| 添加新包 | `uv add <package-name>` |
| 可编辑安装项目 | `uv pip install -e .` |

### 关键警告

运行项目时必须加 `--no-sync`，否则 uv 会自动覆盖自定义的 `tortoise-orm` 版本：

```bash
# ❌ 不要这样
uv run python main.py

# ✅ 正确做法
uv run --no-sync python main.py
```

## 检查清单

完成代码修改后逐项检查：

- [ ] 注释只解释 "为什么"，不解释 "是什么"。
- [ ] 没有为自解释代码（如 `i += 1`、`len(users)`）添加冗余注释。
- [ ] 核心服务类和 API 接口的输入输出参数结构已清晰标注。
- [ ] 只修改了与任务直接相关的文件和代码行。
- [ ] 没有顺手重构无关的历史代码。
- [ ] 没有触发整个文件的重新格式化。
- [ ] 遇到模糊需求时，已向用户提问而非盲目猜测。
- [ ] 提问时提供了带优缺点的选择题。
- [ ] 高危操作（大规模重构、删除核心代码、覆盖重要文件）已征得用户同意。
- [ ] 未明示的需求变更已说明影响并确认。
- [ ] 用户最新指示优先于历史对话上下文。
- [ ] 临时指令（"先跑通"、"写草稿"）已按最高效率执行，未受规范束缚。
- [ ] 所有命令使用 `uv`，运行项目时加了 `--no-sync`。
- [ ] 运行了 Python 语法检查或相关测试。

## 常用验证命令

```bash
# 语法检查
uv run --no-sync python -m py_compile <file>.py

# 运行项目
uv run --no-sync python main.py

# 运行测试
uv run --no-sync python -m pytest
```
